# Technical Spike Plan

## 목적

Phase 1은 완성 UI를 만드는 단계가 아니다. 실제로 자주 받는 CT cine MP4에서 정확한 프레임 인덱스 탐색과 현실적인 캐시 전략이 가능한지 최소 검증판으로 판단한다.

## 선행 조건

- 사용자가 개인정보를 제거했거나 테스트 사용을 승인한 대표 영상 3~5개를 제공한다.
- 샘플은 가능한 범위에서 짧음/김, 고화질/저화질, 카카오톡 일반 전송/파일 전송, H.264/H.265, CFR/VFR 차이를 포함한다.
- 샘플 영상과 분석 출력은 외부 서비스로 전송하지 않는다.
- 기술 구성과 FFmpeg 바이너리 출처·라이선스는 사용자 승인을 받는다.

## 검증 질문

1. ffprobe가 코덱, 해상도, FPS 표기, duration, 추정/실제 프레임 수, 프레임 PTS, 키프레임 정보를 안정적으로 제공하는가
2. presentation order 기준으로 디코딩한 연속 인덱스가 화면 프레임과 일치하는가
3. CFR과 VFR에서 `frameIndex`와 `PTS`를 별도로 관리할 수 있는가
4. 다음 프레임과 이전 프레임을 반복할 때 누락·중복·순서 역전이 없는가
5. 캐시 밖 이전 프레임을 가까운 키프레임부터 재디코딩할 때 체감 지연이 허용 가능한가
6. 전체 선디코딩과 구간 캐시 중 어떤 방식이 대표 영상에 적합한가
7. 한글 파일명·공백·긴 Windows 경로에서 분석, 디코딩, 정리가 정상인가
8. 손상·불완전 영상과 지원하지 않는 코덱에서 앱/검증 도구가 통제된 오류를 내는가

## 실험 단계

### 1. 샘플 목록과 개인정보 확인

- 파일은 앱 작업용 로컬 경로에서 읽기 전용으로 취급한다.
- 원본 파일명 자체에 개인정보가 있으면 보고서에는 익명 ID를 사용한다.
- 파일 크기, 해상도, duration을 기록하되 환자 식별 정보는 기록하지 않는다.

### 2. ffprobe 분석

각 샘플에서 다음 값을 수집한다.

- container와 video codec/profile/pixel format
- width, height, rotation/SAR/DAR
- `avg_frame_rate`, `r_frame_rate`, duration
- 컨테이너 표기 프레임 수와 실제 count 결과
- 프레임별 PTS/time, duration, keyframe 여부
- PTS 누락·중복·비단조 여부

### 3. 순차 디코딩 검증

- 디코딩된 각 프레임에 0부터 연속 인덱스를 부여한다.
- 테스트용으로 프레임 수와 PTS 목록을 기록한다.
- 선택 지점의 이미지 hash 또는 비교 가능한 표식을 사용해 왕복 탐색 동일성을 확인한다.

### 4. 캐시 전략 비교

#### A. 전체 프레임 선디코딩

- 첫 화면 우선 표시 후 나머지 프레임을 처리하는 방식과 전체 완료 후 표시하는 방식을 구분한다.
- 메모리 raw frame, 임시 무손실/고품질 이미지, 혼합 방식을 비교한다.

#### B. 구간 캐시

- 현재 프레임 전후 고정 개수 또는 메모리 예산 기반 LRU/ring buffer를 비교한다.
- 캐시 miss 이전 탐색에서 keyframe부터 목표 프레임까지 재디코딩한다.

측정값:

- 첫 프레임 표시 시간
- 순차 전진/후진 및 캐시 miss 지연의 p50/p95/최대
- peak memory와 임시 디스크 사용량
- 전체 선디코딩 완료 시간
- 프로세스 종료와 캐시 정리 시간

### 5. Windows 경로·오류·정리

- 한글, 공백, 괄호, 긴 경로 샘플 복사본으로 인자 전달을 검증한다.
- 영상 중간 잘림과 잘못된 확장자를 별도 테스트한다.
- 정상 종료, 강제 종료 모사, 다음 실행 정리에서 앱 전용 캐시 밖 파일이 삭제되지 않는지 확인한다.

## 합격 기준 초안

- 모든 정상 샘플에서 디코딩 인덱스가 0부터 연속이며 왕복 탐색 비교에 불일치가 없다.
- VFR 샘플에서 프레임 인덱스와 PTS가 분리 저장되고 시간 계산 때문에 프레임이 건너뛰지 않는다.
- 대표 기준 영상에서 첫 프레임 2초 이내, 캐시 hit 이동 약 50ms 이내를 목표로 하며 실제 수치와 환경을 기록한다.
- 캐시 miss 지연이 목표를 넘더라도 원인과 사용자에게 보일 로딩 상태를 정량화할 수 있다.
- peak memory·disk 수치로 기본 캐시 전략을 선택할 증거가 생긴다.
- 오류 입력이 검증판 전체를 종료시키지 않고 원본을 변경하지 않는다.
- 정상·비정상 종료 뒤 앱 전용 임시 파일만 정리된다.

## 실패·중단 기준

- 대표 MP4에서 프레임 순서나 동일성 검증이 반복해서 깨진다.
- 선택한 전달 방식이 메모리 또는 IPC 병목으로 목표에 크게 못 미치며 단순한 대안이 없다.
- FFmpeg 배포·라이선스 조건을 만족할 빌드를 확정할 수 없다.
- 샘플이 현실 사용을 대표하지 못해 캐시 결정을 내릴 근거가 없다.
- 개인정보가 제거되지 않은 샘플을 승인 없이 분석해야만 진행할 수 있다.

실패 시 기능 구현으로 넘어가지 않고 원인, 재현 절차, 대안 후보를 사용자에게 보고한다.

## 산출물

- 익명화된 샘플 특성표
- 재현 가능한 분석·측정 절차
- 프레임 인덱스/PTS 일관성 결과
- 캐시 전략 비교표와 1순위 제안
- 확인된 실패 사례와 남은 위험
- Phase 2 진입 여부에 대한 사용자 승인 요청

## 이번 문서에서 확정하지 않는 것

- 전체 선디코딩 또는 구간 캐시의 최종 선택
- 고정 캐시 프레임 수
- 지원 최대 해상도·프레임 수·파일 크기
- 특정 FFmpeg 배포판과 라이선스 구성
- 완성 앱 UI와 패키징 방식
